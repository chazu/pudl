# B1.2 — Manifest Ingestion

**Epic:** E1 — Close the ACUTE Feedback Loop
**Blocked by:** nothing (can parallelize with B1.1)
**Blocks:** B1.3 (status tracking)

## Summary

Add a `pudl ingest-manifest` command that accepts mu build manifest output and records per-action results in the catalog, so pudl knows what mu did and whether it succeeded.

## Background

- mu build can emit a manifest JSON document describing all actions executed
- The manifest matches `pudl/mu.#Manifest` (defined in `pudl/mu/mu.cue`)
- `#Manifest` has identity field `timestamp` and tracked fields `summary`, `actions`
- Each action in the manifest has: `id`, `cached` (bool), `exit_code` (int), `outputs` (map)
- The catalog already has `run_id`, `definition`, `method` fields for artifact tracking

## Implementation

### 1. New file: `internal/mubridge/manifest.go`

```go
package mubridge

// ManifestInput represents a mu build manifest.
type ManifestInput struct {
    Timestamp string           `json:"timestamp"`
    Summary   ManifestSummary  `json:"summary"`
    Actions   []ManifestAction `json:"actions"`
}

type ManifestSummary struct {
    Total    int `json:"total"`
    Cached   int `json:"cached"`
    Executed int `json:"executed"`
    Failed   int `json:"failed"`
}

type ManifestAction struct {
    ID       string            `json:"id"`
    Target   string            `json:"target,omitempty"`
    Cached   bool              `json:"cached"`
    ExitCode int               `json:"exit_code"`
    Outputs  map[string]string `json:"outputs,omitempty"`
}

// IngestManifest processes a mu build manifest and stores results in the catalog.
// Returns the run_id assigned to this manifest and any error.
func IngestManifest(db *database.CatalogDB, reader io.Reader, origin string) (string, error)
```

Logic:
- Read entire JSON from `reader`, unmarshal into `ManifestInput`
- Generate a `run_id` from the manifest timestamp + content hash (deterministic)
- Store the full manifest as a single catalog entry:
  - `EntryType`: `"manifest"`
  - `Origin`: `origin` (default `"mu-build"`)
  - `Schema`: `"pudl/mu.#Manifest"`
  - `RunID`: generated run_id
  - `Format`: `"json"`
- For each action in `Actions`:
  - Extract target name (strip `//`) to get the definition name
  - Create an additional catalog entry per action:
    - `EntryType`: `"manifest-action"`
    - `Definition`: target/definition name
    - `RunID`: same run_id
    - `Tags`: JSON-encoded `{"exit_code": N, "cached": true/false}`
    - `Schema`: `"pudl/mu.#ManifestAction"`
  - Store action JSON in raw data directory
- Content-hash dedup: skip if manifest with same content hash already ingested

### 2. New file: `cmd/ingest_manifest.go`

```go
var ingestManifestCmd = &cobra.Command{
    Use:   "ingest-manifest",
    Short: "Ingest mu build manifest into the catalog",
    Long: `Read a mu build manifest (JSON) and store action results in the catalog.
This records what mu did during convergence, enabling status tracking.

Examples:
    mu build --emit-manifest | pudl ingest-manifest
    pudl ingest-manifest --path manifest.json`,
    RunE: func(cmd *cobra.Command, args []string) error { ... },
}
```

Flags:
- `--path <file>` — read from file instead of stdin
- `--origin <string>` — override origin (default: `"mu-build"`)

Logic:
- If `--path` is set, open file; otherwise use `os.Stdin`
- Call `mubridge.IngestManifest(db, reader, origin)`
- Print summary: "Ingested manifest (run_id: X): N actions (M cached, K failed)"

### 3. Modify: `internal/database/catalog.go`

Add query methods:

```go
// GetManifestActions returns all manifest-action entries for a given run_id.
func (db *CatalogDB) GetManifestActions(runID string) ([]CatalogEntry, error)

// GetLatestManifestAction returns the most recent manifest-action for a definition.
func (db *CatalogDB) GetLatestManifestAction(definitionName string) (*CatalogEntry, error)
```

SQL for `GetLatestManifestAction`:
```sql
SELECT ... FROM catalog_entries
WHERE entry_type = 'manifest-action' AND definition = ?
ORDER BY import_timestamp DESC LIMIT 1
```

No migration needed — reuses existing columns (`entry_type`, `definition`, `run_id`, `tags`).

## Tests

Create `internal/mubridge/manifest_test.go`:

1. **TestIngestManifest_Basic** — ingest a manifest with 3 actions (1 cached, 1 success, 1 failed), verify:
   - 1 manifest entry + 3 manifest-action entries created
   - All share the same run_id
   - Tags contain correct exit_code and cached values
2. **TestIngestManifest_Dedup** — ingest same manifest twice, verify only one set of entries
3. **TestIngestManifest_EmptyActions** — manifest with no actions creates only the manifest entry

## Acceptance Criteria

- `pudl ingest-manifest --path manifest.json` creates catalog entries for the manifest and each action
- `pudl list --schema pudl/mu.#Manifest` shows the manifest entry
- `pudl list --schema pudl/mu.#ManifestAction` shows per-action entries
- Entries are grouped by `run_id`
- Tags on manifest-action entries contain `exit_code` and `cached` fields
- Duplicate manifests (same content hash) are skipped
